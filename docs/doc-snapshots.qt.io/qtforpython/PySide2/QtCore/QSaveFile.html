Detailed Description¶

The PySide2.QtCore.QSaveFile class provides an interface for safely writing to files.

PySide2.QtCore.QSaveFile is an I/O device for writing text and binary files, without losing existing data if the writing operation fails.

While writing, the contents will be written to a temporary file, and if no error happened, PySide2.QtCore.QSaveFile.commit() will move it to the final file. This ensures that no data at the final file is lost in case an error happens while writing, and no partially-written file is ever present at the final location. Always use PySide2.QtCore.QSaveFile when saving entire documents to disk.

PySide2.QtCore.QSaveFile automatically detects errors while writing, such as the full partition situation, where PySide2.QtCore.QIODevice.write() cannot write all the bytes. It will remember that an error happened, and will discard the temporary file in PySide2.QtCore.QSaveFile.commit() .

Much like with PySide2.QtCore.QFile , the file is opened with PySide2.QtCore.QSaveFile.open() . Data is usually read and written using PySide2.QtCore.QDataStream or PySide2.QtCore.QTextStream , but you can also call the PySide2.QtCore.QIODevice -inherited functions PySide2.QtCore.QIODevice.read() , PySide2.QtCore.QIODevice.readLine() , PySide2.QtCore.QIODevice.readAll() , PySide2.QtCore.QIODevice.write() .

Unlike PySide2.QtCore.QFile , calling PySide2.QtCore.QSaveFile.close() is not allowed. PySide2.QtCore.QSaveFile.commit() replaces it. If PySide2.QtCore.QSaveFile.commit() was not called and the PySide2.QtCore.QSaveFile instance is destroyed, the temporary file is discarded.

To abort saving due to an application error, call PySide2.QtCore.QSaveFile.cancelWriting() , so that even a call to PySide2.QtCore.QSaveFile.commit() later on will not save.

PySide2.QtCore.QSaveFile.cancelWriting()¶

Cancels writing the new file.

If the application changes its mind while saving, it can call , which sets an error code so that PySide2.QtCore.QSaveFile.commit() will discard the temporary file.

Alternatively, it can simply make sure not to call PySide2.QtCore.QSaveFile.commit() .

Further write operations are possible after calling this method, but none of it will have any effect, the written file will be discarded.

This method has no effect when direct write fallback is used. This is the case when saving over an existing file in a readonly directory: no temporary file can be created, so the existing file is overwritten no matter what, and cannot do anything about that, the contents of the existing file will be lost.

PySide2.QtCore.QSaveFile.commit()¶
Return type:PySide2.QtCore.bool

Commits the changes to disk, if all previous writes were successful.

It is mandatory to call this at the end of the saving operation, otherwise the file will be discarded.

If an error happened during writing, deletes the temporary file and returns false . Otherwise, renames it to the final PySide2.QtCore.QSaveFile.fileName() and returns true on success. Finally, closes the device.

PySide2.QtCore.QSaveFile.setDirectWriteFallback(enabled)¶
Parameters:enabled – PySide2.QtCore.bool

Allows writing over the existing file if necessary.

PySide2.QtCore.QSaveFile creates a temporary file in the same directory as the final file and atomically renames it. However this is not possible if the directory permissions do not allow creating new files. In order to preserve atomicity guarantees, PySide2.QtCore.QSaveFile.open() fails when it cannot create the temporary file.

In order to allow users to edit files with write permissions in a directory with restricted permissions, call with enabled set to true, and the following calls to PySide2.QtCore.QSaveFile.open() will fallback to opening the existing file directly and writing into it, without the use of a temporary file. This does not have atomicity guarantees, i.e. an application crash or for instance a power failure could lead to a partially-written file on disk. It also means PySide2.QtCore.QSaveFile.cancelWriting() has no effect, in such a case.

Typically, to save documents edited by the user, call (true), and to save application internal files (configuration files, data files, …), keep the default setting which ensures atomicity.
