Detailed Description¶

The PySide2.QtCore.QFile class provides an interface for reading from and writing to files.

PySide2.QtCore.QFile is an I/O device for reading and writing text and binary files and resources . A PySide2.QtCore.QFile may be used by itself or, more conveniently, with a PySide2.QtCore.QTextStream or PySide2.QtCore.QDataStream .

The file name is usually passed in the constructor, but it can be set at any time using PySide2.QtCore.QFile.setFileName() . PySide2.QtCore.QFile expects the file separator to be ‘/’ regardless of operating system. The use of other separators (e.g., ‘') is not supported.

You can check for a file’s existence using PySide2.QtCore.QFile.exists() , and remove a file using PySide2.QtCore.QFile.remove() . (More advanced file system related operations are provided by PySide2.QtCore.QFileInfo and PySide2.QtCore.QDir .)

The file is opened with PySide2.QtCore.QFile.open() , closed with PySide2.QtCore.QFileDevice.close() , and flushed with PySide2.QtCore.QFileDevice.flush() . Data is usually read and written using PySide2.QtCore.QDataStream or PySide2.QtCore.QTextStream , but you can also call the PySide2.QtCore.QIODevice -inherited functions PySide2.QtCore.QIODevice.read() , PySide2.QtCore.QIODevice.readLine() , PySide2.QtCore.QIODevice.readAll() , PySide2.QtCore.QIODevice.write() . PySide2.QtCore.QFile also inherits PySide2.QtCore.QIODevice.getChar() , PySide2.QtCore.QIODevice.putChar() , and PySide2.QtCore.QIODevice.ungetChar() , which work one character at a time.

The size of the file is returned by PySide2.QtCore.QFile.size() . You can get the current file position using PySide2.QtCore.QFileDevice.pos() , or move to a new file position using PySide2.QtCore.QFileDevice.seek() . If you’ve reached the end of the file, PySide2.QtCore.QFileDevice.atEnd() returns true .

Reading Files Directly¶

The following example reads a text file line by line:

file = QFile("in.txt")
if not file.open(QIODevice.ReadOnly | QIODevice.Text):
    return

while not file.atEnd():
    line = file.readLine() # A QByteArray
    process_line(line)

The QIODevice.Text flag passed to PySide2.QtCore.QFile.open() tells Qt to convert Windows-style line terminators (“\r\n”) into C++-style terminators (“\n”). By default, PySide2.QtCore.QFile assumes binary, i.e. it doesn’t perform any conversion on the bytes stored in the file.

Using Streams to Read Files¶

The next example uses PySide2.QtCore.QTextStream to read a text file line by line:

file = QFile("in.txt")
if not file.open(QIODevice.ReadOnly | QIODevice.Text):
    return

in = QTextStream(file)
while not in.atEnd():
    line = in.readLine() # A QByteArray
    process_line(line)

PySide2.QtCore.QTextStream takes care of converting the 8-bit data stored on disk into a 16-bit Unicode PySide2.QtCore.QString . By default, it assumes that the user system’s local 8-bit encoding is used (e.g., UTF-8 on most unix based operating systems; see QTextCodec.codecForLocale() for details). This can be changed using QTextStream.setCodec() .

To write text, we can use operator<<(), which is overloaded to take a PySide2.QtCore.QTextStream on the left and various data types (including PySide2.QtCore.QString ) on the right:

file = QFile("out.txt")
if not file.open(QIODevice.WriteOnly | QIODevice.Text):
    return

out = QTextStream(file)
out << "The magic number is: " << 49 << "\n"

PySide2.QtCore.QDataStream is similar, in that you can use operator<<() to write data and operator>>() to read it back. See the class documentation for details.

When you use PySide2.QtCore.QFile , PySide2.QtCore.QFileInfo , and PySide2.QtCore.QDir to access the file system with Qt, you can use Unicode file names. On Unix, these file names are converted to an 8-bit encoding. If you want to use standard C++ APIs (<cstdio> or <iostream> ) or platform-specific APIs to access files instead of PySide2.QtCore.QFile , you can use the PySide2.QtCore.QFile.encodeName() and PySide2.QtCore.QFile.decodeName() functions to convert between Unicode file names and 8-bit file names.

On Unix, there are some special system files (e.g. in /proc ) for which PySide2.QtCore.QFile.size() will always return 0, yet you may still be able to read more data from such a file; the data is generated in direct response to you calling PySide2.QtCore.QIODevice.read() . In this case, however, you cannot use PySide2.QtCore.QFileDevice.atEnd() to determine if there is more data to read (since PySide2.QtCore.QFileDevice.atEnd() will return true for a file that claims to have size 0). Instead, you should either call PySide2.QtCore.QIODevice.readAll() , or call PySide2.QtCore.QIODevice.read() or PySide2.QtCore.QIODevice.readLine() repeatedly until no more data can be read. The next example uses PySide2.QtCore.QTextStream to read /proc/modules line by line:

file = QFile("/proc/modules")
if not file.open(QIODevice.ReadOnly | QIODevice.Text):
    return

in = QTextStream(file)
line = in.readLine()
while not line.isNull():
    process_line(line)
    line = in.readLine()

Signals¶

Unlike other PySide2.QtCore.QIODevice implementations, such as PySide2.QtNetwork.QTcpSocket , PySide2.QtCore.QFile does not emit the PySide2.QtCore.QIODevice.aboutToClose() , PySide2.QtCore.QIODevice.bytesWritten() , or PySide2.QtCore.QIODevice.readyRead() signals. This implementation detail means that PySide2.QtCore.QFile is not suitable for reading and writing certain types of files, such as device files on Unix platforms.

Platform Specific Issues¶

File permissions are handled differently on Unix-like systems and Windows. In a non writable directory on Unix-like systems, files cannot be created. This is not always the case on Windows, where, for instance, the ‘My Documents’ directory usually is not writable, but it is still possible to create files in it.

Qt’s understanding of file permissions is limited, which affects especially the QFile.setPermissions() function. On Windows, Qt will set only the legacy read-only flag, and that only when none of the Write* flags are passed. Qt does not manipulate access control lists (ACLs), which makes this function mostly useless for NTFS volumes. It may still be of use for USB sticks that use VFAT file systems. POSIX ACLs are not manipulated, either.

PySide2.QtCore.QFile.link(newName)¶
Parameters:newName – unicode
Return type:PySide2.QtCore.bool

Creates a link named linkName that points to the file currently specified by PySide2.QtCore.QFile.fileName() . What a link is depends on the underlying filesystem (be it a shortcut on Windows or a symbolic link on Unix). Returns true if successful; otherwise returns false .

This function will not overwrite an already existing entity in the file system; in this case, link() will return false and set PySide2.QtCore.QFile.error() to return RenameError .

Note

To create a valid link on Windows, linkName must have a .lnk file extension.

PySide2.QtCore.QFile.open(fd, ioFlags[, handleFlags=QFileDevice.DontCloseHandle])¶
Parameters:
  • fd – PySide2.QtCore.int
  • ioFlags – PySide2.QtCore.QIODevice.OpenMode
  • handleFlags – PySide2.QtCore.QFileDevice.FileHandleFlags
Return type:

PySide2.QtCore.bool

This is an overloaded function.

Opens the existing file descriptor fd in the given mode . handleFlags may be used to specify additional options. Returns true if successful; otherwise returns false .

When a PySide2.QtCore.QFile is opened using this function, behaviour of PySide2.QtCore.QFileDevice.close() is controlled by the AutoCloseHandle flag. If AutoCloseHandle is specified, and this function succeeds, then calling PySide2.QtCore.QFileDevice.close() closes the adopted handle. Otherwise, PySide2.QtCore.QFileDevice.close() does not actually close the file, but only flushes it.

The PySide2.QtCore.QFile that is opened using this function is automatically set to be in raw mode; this means that the file input/output functions are slow. If you run into performance issues, you should try to use one of the other open functions.

Warning

If fd is not a regular file, e.g, it is 0 (stdin ), 1 (stdout ), or 2 (stderr ), you may not be able to PySide2.QtCore.QFileDevice.seek() . In those cases, PySide2.QtCore.QFile.size() returns 0 . See QIODevice.isSequential() for more information.

Warning

Since this function opens the file without specifying the file name, you cannot use this PySide2.QtCore.QFile with a PySide2.QtCore.QFileInfo .

PySide2.QtCore.QFile.rename(newName)¶
Parameters:newName – unicode
Return type:PySide2.QtCore.bool

Renames the file currently specified by PySide2.QtCore.QFile.fileName() to newName . Returns true if successful; otherwise returns false .

If a file with the name newName already exists, returns false (i.e., PySide2.QtCore.QFile will not overwrite it).

The file is closed before it is renamed.

If the rename operation fails, Qt will attempt to copy this file’s contents to newName , and then remove this file, keeping only newName . If that copy operation fails or this file can’t be removed, the destination file newName is removed to restore the old state.

PySide2.QtCore.QFile.setFileName(name)¶
Parameters:name – unicode

Sets the name of the file. The name can have no path, a relative path, or an absolute path.

Do not call this function if the file has already been opened.

If the file name has no path or a relative path, the path used will be the application’s current directory path at the time of the PySide2.QtCore.QFile.open() ** call.

Example:

file = QFile()
QDir.setCurrent("/tmp")
file.setFileName("readme.txt")
QDir.setCurrent("/home")
file.open(QIODevice.ReadOnly)       # opens "/home/readme.txt" under Unix

Note that the directory separator “/” works for all operating systems supported by Qt.
